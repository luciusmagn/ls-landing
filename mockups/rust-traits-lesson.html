Traits: shared behavior without inheritance

A trait names a capability. Types opt in by implementing it, the compiler enforces the contract, and nothing inherits anything. This lesson builds the emulator's debug vocabulary with one small trait and follows it from plain calls through generics to trait objects.

A trait declares a capability

Our emulator wants to print things on its trace console: opcodes, registers, cartridge headers. The capability is "can describe itself", so that is exactly what we declare. The trait is the contract; the impl block is a type's signature under that contract. 

/// Anything the debugger can print on the trace console.
pub trait Describe {
    fn describe(&self) -> String;
}

pub struct Opcode {
    pub byte: u8,
    pub cycles: u8,
}

impl Describe for Opcode {
    fn describe(&self) -> String {
        format!("{:#04x} ({} cycles)", self.byte, self.cycles)
    }
}

Traits are nominal: a type does not satisfy Describe by accident of having the right method, it satisfies it because an impl says so. You may implement your traits for types you did not write, and foreign traits for your types; the one combination the compiler refuses is a foreign trait for a foreign type. That coherence rule is why two crates can never disagree about what Opcode::describe does.

Default methods

A trait may ship method bodies. Implementors get them for free and override only what they must, which keeps the common case silent. 

pub trait Cartridge {
    fn read(&self, addr: u16) -> u8;
    fn write(&mut self, addr: u16, value: u8);

    /// Most cartridges have exactly one ROM bank; let them say nothing.
    fn bank_count(&self) -> usize {
        1
    }
}

In the project, only the banked mappers override bank_count. The ROM-only cartridge from lesson 12 implements two methods and is done.

Trait bounds

Generic code names its requirements as bounds. The function below works for any describable type, and the compiler generates a separate, fully concrete copy per type it is used with. That is monomorphization: the abstraction is gone by the time the optimizer sees the code, so a bound costs nothing at run time. 

fn trace<T: Describe>(step: u32, item: &T) {
    eprintln!("[{step:06}] {}", item.describe());
}

// The same requirement, spelled with a where clause:
fn trace_all<I>(items: I)
where
    I: IntoIterator,
    I::Item: Describe,
{
    for (step, item) in items.into_iter().enumerate() {
        eprintln!("[{step:06}] {}", item.describe());
    }
}

Reach for the angle-bracket form while the bounds fit on one line and the where clause the moment they do not. The two mean the same thing; one of them stays readable.

Trait objects

Sometimes the set of types is only known at run time: the debugger watches a mixed list of opcodes, registers, and mapper state. dyn Describe erases the concrete type behind a pointer and a vtable, trading a call indirection for the ability to mix. 

let watchlist: Vec<Box<dyn Describe>> = vec![
    Box::new(Opcode { byte: 0x3e, cycles: 8 }),
    Box::new(Register16 { name: "HL", value: 0x9fff }),
];

for entry in &watchlist {
    println!("{}", entry.describe());
}

Not every trait can do this: to be dyn-compatible (the rule formerly called object safety), a trait must avoid generic methods and methods returning bare Self, because a vtable cannot answer for a type it no longer knows.

Choosing a dispatch
Question Generics, T: Describe Trait objects, dyn Describe
When is the method resolved? Compile time, monomorphized Run time, through a vtable
Call cost None; inlining applies One pointer indirection
Binary size One copy of the code per type One copy total
Mixed collections No, every element is the same type Yes, Vec<Box<dyn Describe>>
Trait requirements Any trait Dyn-compatible traits only
